## Summary

Fas 04.3 består av tre separata och oberoende uppgifter: (1) implementera deep link från extremväder-push till WeatherDetailSheet, (2) lägga till fcmToken i AppUser Codable-modellen, och (3) städa bort dead code och debug-prints. Alla tre är väl avgränsade och kan implementeras utan beroenden av varandra.

Deep link-implementationen är den tekniskt mest intressanta uppgiften. FCM-payloaden är redan klar — `weatherAlertTrigger.ts` skickar `type: "weatherAlert"`, `friendId`, `friendName` och `friendCity`. Det som saknas är iOS-sidan: AppDelegate fångar inte upp `type: "weatherAlert"` alls, och det finns ingen navigationskedja från push till WeatherDetailSheet. Befintligt mönster (chat deep link) ger en klar mall att följa, med en viktig skillnad: väderdetalj-destinationen kräver ett `FriendWeather`-objekt som måste konstrueras från push-payloadens `friendId`.

fcmToken-problemet är enkelt men viktigt: AppUser.swift saknar `fcmToken`-propertyn vilket innebär att varje gång AppUser-data skrivs till Firestore med `.setData()` eller `.updateData()` — exempelvis vid profiluppdatering — riskerar fcmToken-fältet att raderas. Lösningen är en trivial Codable-tillägg.

**Primary recommendation:** Följ exakt samma NotificationCenter-mönster som chat deep link, men skicka `friendId` istället för `conversationId`. FriendListView konstruerar FriendWeather lokalt från sina befintliga `favorites + others`-arrayer om vännen redan är laddad, annars fetchar den direkt.

## Architecture Patterns

### Recommended Navigation Chain

```
UNNotificationResponse (AppDelegate)
  ↓ NotificationCenter.post(.openWeatherAlert, object: friendId)
AppRouter.onReceive(.openWeatherAlert)
  ↓ sets @State var openWeatherAlertFriendId: String?
MainTabView (tar emot som @Binding)
  ↓ byter till Vänner-tab (TabView selection)
FriendListView
  ↓ onChange(of: openWeatherAlertFriendId) → konstruerar FriendWeather → sätter selectedFriendWeather
WeatherDetailSheet (befintlig) visas som .sheet
```

### Pattern 1: Befintlig chat deep link (mall att följa)

**Vad:** AppDelegate fångar push-tap → postar NotificationCenter → AppRouter lyssnar → sätter binding → ConversationListView läser binding.

**Exakt implementation (från codebase):**

```swift
// AppDelegate.swift — didReceive
let userInfo = response.notification.request.content.userInfo
if let conversationId = userInfo["conversationId"] as? String {
    NotificationCenter.default.post(name: .openChat, object: conversationId)
}

// AppRouter.swift — lyssnar
.onReceive(NotificationCenter.default.publisher(for: .openChat)) { notification in
    if let conversationId = notification.object as? String {
        openConversationId = conversationId
    }
}

// ConversationListView.swift — reagerar
.onChange(of: openConversationId) { _, newId in
    if let id = newId {
        navigationPath.append(id)
        openConversationId = nil  // reset
    }
}
```

**Pattern 2: Weather deep link (ny implementation)**

```swift
// AppDelegate.swift — utöka didReceive
if let type = userInfo["type"] as? String, type == "weatherAlert",
   let friendId = userInfo["friendId"] as? String {
    NotificationCenter.default.post(name: .openWeatherAlert, object: friendId)
}

// AppRouter.swift — ny Notification.Name + listener
extension Notification.Name {
    static let openWeatherAlert = Notification.Name("openWeatherAlert")
}

@State private var openWeatherAlertFriendId: String? = nil

// I authenticated case:
MainTabView(
    openConversationId: $openConversationId,
    openWeatherAlertFriendId: $openWeatherAlertFriendId
)
.onReceive(NotificationCenter.default.publisher(for: .openWeatherAlert)) { notification in
    if let friendId = notification.object as? String {
        openWeatherAlertFriendId = friendId
    }
}

// MainTabView.swift — ny binding + tab selection
struct MainTabView: View {
    @Binding var openConversationId: String?
    @Binding var openWeatherAlertFriendId: String?
    @State private var selectedTab = 0  // 0 = Vänner, 1 = Chattar

    // TabView(selection: $selectedTab) — byt till 0 när openWeatherAlertFriendId sätts
}
```

### Pattern 3: FriendListView - konstruera FriendWeather från friendId

**Utmaning:** WeatherDetailSheet kräver ett `FriendWeather`-objekt med `friend` + `weather`. Chat-mönstret skickar bara ett ID och navigerar till en vy som hanterar data-loading själv. FriendListView måste hitta rätt FriendWeather.

**Lösning — sök i befintliga arrayer:**
```swift
// FriendListView.swift
.onChange(of: openWeatherAlertFriendId) { _, friendId in
    guard let friendId else { return }
    // Sök i befintliga favorites + others
    let all = viewModel.favorites + viewModel.others
    if let fw = all.first(where: { $0.friend.id == friendId }) {
        selectedFriendWeather = fw  // WeatherDetailSheet öppnas automatiskt
    }
    openWeatherAlertFriendId = nil  // reset
}
```

**Fallback om vännen inte är laddad:** Konstruera ett minimalt FriendWeather från push-data (friendName, friendCity från payload) utan weather-data. WeatherDetailSheet hanterar redan `weather == nil`-fallet med ett felmeddelande.

**Cold start-beslut (Claude's Discretion):** Rekommendation — stöd INTE cold start i v1. Anledning: Cold start kräver att appen börjar ladda friend + weather data *innan* navigationsträdet är redo, vilket ökar komplexiteten avsevärt. Chat-tryckknappar stöder heller inte cold start. Räcker att hantera foreground och background-tap.

**Förgrundsbeteende (Claude's Discretion):** AppDelegate låter redan `.list, .banner, .sound` visas i förgrunden (willPresent). Banner visas men navigerar inte automatiskt — användaren väljer att trycka. Inget extra att göra.

**Alert-banner (Claude's Discretion):** Inga extra alert-banner i WeatherDetailSheet — befintlig väderdetalj räcker. Alert-info (alertSummary) finns i Friend-modellen och kan visas i framtida fas om det behövs.

### Pattern 4: TabView selection binding

**Problem:** MainTabView behöver kunna byta till Vänner-tabben (index 0) programmatiskt när openWeatherAlertFriendId sätts.

```swift
// MainTabView.swift
@State private var selectedTab = 0

TabView(selection: $selectedTab) {
    NavigationStack {
        FriendListView(openWeatherAlertFriendId: $openWeatherAlertFriendId)
    }
    .tag(0)
    .tabItem { Label("Vänner", systemImage: "person.2") }

    NavigationStack {
        ConversationListView(openConversationId: $openConversationId)
    }
    .tag(1)
    .tabItem { Label("Chattar", systemImage: "bubble.left.and.bubble.right") }
}
.onChange(of: openWeatherAlertFriendId) { _, id in
    if id != nil { selectedTab = 0 }
}
```

**Notera:** FriendListView tar idag inga bindings — den behöver en ny `@Binding var openWeatherAlertFriendId: String?` parameter. Alternativt kan binding hållas i MainTabView och skickas vidare via miljö. Binding är enklare och matchar befintligt ConversationListView-mönster.

### Anti-Patterns to Avoid

- **Cold start deep link i v1:** Kräver state-maskin för "pending navigation" som komplicerar appens start-sekvens avsevärt. Inte värt det.
- **Skicka FriendWeather i NotificationCenter.object:** FriendWeather är en struct med WeatherKit-data — inte serialiserbar. Skicka alltid bara friendId (String).
- **Återanvända openChat notification:** Semantiskt fel. Type-checken i AppDelegate (`userInfo["type"]`) räcker för att separera push-typer, men NotificationCenter-names ska vara separata.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tab switching | Custom tab manager | SwiftUI `TabView(selection:)` med `@State` | Inbyggt och stabilt |
| Push payload parsing | Custom deserializer | Direkt dictionary lookup på `userInfo` | FCM `data`-dict är alltid `[AnyHashable: Any]` |
| FriendWeather utan weather | Komplex lazy-loading | Konstruera FriendWeather(friend:, weather: nil) | WeatherDetailSheet hanterar nil-case redan |

## Common Pitfalls

### Pitfall 1: fcmToken raderas vid AppUser.setData()
**What goes wrong:** AppUser.swift saknar `fcmToken`-property. Om kod någonstans skriver ett AppUser-objekt via `.setData()` (ej `.updateData()`) till `users/{uid}` utan `merge: true`, raderas fcmToken-fältet i Firestore.

**Why it happens:** Codable-encoding av AppUser inkluderar inte fcmToken → fältet saknas i dict → setData raderar det.

**How to avoid:** Lägg till `var fcmToken: String?` i AppUser. Codable-encoding ignorerar nil-optionals som standard (om `encodeIfPresent` används, vilket FirebaseFirestore gör).

**Verifiering:** Kontrollera att OnboardingViewModel.swift (som använder `.setData()` direkt) inte skriver AppUser-typ utan att behålla fcmToken.

**Notera från STATE.md:** `[Phase 01-03]: OnboardingViewModel skriver direkt till Firestore via .document(uid).setData() — UserService.createUserProfile kräver @DocumentID som inte kan sättas vid skapande`. Detta bekräftar att setData används — fcmToken-skydd är nödvändigt.

### Pitfall 2: FriendListView hittar inte vän vid push-tap
**What goes wrong:** Om push-tap sker och FriendListView inte har hunnit ladda vänner (t.ex. snabb cold start), finns ingen matchning i `favorites + others`.

**Why it happens:** Laddning är asynkron — viewModel.load() har inte körts klart.

**How to avoid:** Hantera gracefully — om `all.first(where: { $0.friend.id == friendId })` returnerar nil, logga och ignorera (inga crashes). Användaren ser Vänner-listan, som laddas strax.

**Alternative:** Lagra `pendingWeatherAlertFriendId` i FriendListViewModel och kolla vid load-completion — överdimensionerat för v1.

### Pitfall 3: MainTabView-binding orsakar extra rendering
**What goes wrong:** Att lägga till `@Binding var openWeatherAlertFriendId: String?` till FriendListView kan orsaka onödiga re-renders av hela vyn om binding ändras.

**Why it happens:** SwiftUI re-renderar views vars bindings ändras.

**How to avoid:** Reset binding direkt i onChange (sätt till nil) — exakt som ConversationListView gör med openConversationId.

### Pitfall 4: Debug-print på rad 125 FriendListViewModel är värdefull
**What goes wrong:** Rad 125 är `print("⚠️ WeatherKit fel för ...")` — ett error-meddelande, inte ett debug-print i samma klass som rad 33.

**Why it happens:** CONTEXT.md anger "FriendListViewModel rad 33 och 125" — men rad 125 är ett felmeddelande i en catch-block.

**How to avoid:** Ta bort både raderna per CONTEXT.md beslut. Felhanteringen i catch-blocket fortsätter att fungera korrekt utan print — det returnerar redan FriendWeather(friend:, weather: nil).

## Code Examples

### fcmToken i AppUser
```swift
// AppUser.swift — lägg till efter authProvider
var fcmToken: String?
```

Codable-egenskapen blir automatiskt optional i encoding. Firestore FirestoreSwift-integration hanterar `nil` som "skip fält vid encoding" — detta är korrekt beteende.

### Ny Notification.Name
```swift
// AppRouter.swift — utöka extension
extension Notification.Name {
    static let openChat = Notification.Name("openChat")
    static let openWeatherAlert = Notification.Name("openWeatherAlert")
}
```

### AppDelegate.didReceive — utökt
```swift
func userNotificationCenter(
    _ center: UNUserNotificationCenter,
    didReceive response: UNNotificationResponse,
    withCompletionHandler completionHandler: @escaping () -> Void
) {
    let userInfo = response.notification.request.content.userInfo
    if let conversationId = userInfo["conversationId"] as? String {
        NotificationCenter.default.post(name: .openChat, object: conversationId)
    } else if let type = userInfo["type"] as? String, type == "weatherAlert",
              let friendId = userInfo["friendId"] as? String {
        NotificationCenter.default.post(name: .openWeatherAlert, object: friendId)
    }
    completionHandler()
}
```

## Open Questions

1. **FirebaseStorage-import i ContactImportService**
   - What we know: `import FirebaseStorage` finns på rad 2. `uploadContactPhoto` är den enda metod som använder `Storage.storage()` och `storage`-propertyn på rad 112.
   - What's unclear: Om `storage` (rad 112) räknas som del av `uploadContactPhoto` — ja, propertyn deklareras på klass-nivå och kan tas bort om ingen annan metod refererar till Storage.
   - Recommendation: Ta bort `private let storage = Storage.storage()` (rad 112) och `import FirebaseStorage` (rad 2) när uploadContactPhoto tas bort. Kompilera för att verifiera att inga kompilatorfel uppstår.

2. **FriendListView — binding-approach vs environment**
   - What we know: ConversationListView tar `@Binding var openConversationId: String?` direkt. FriendListView tar inga bindings idag.
   - What's unclear: Är det bättre att lägga openWeatherAlertFriendId i AppRouter (@State) och skicka som @Binding hela vägen ner, eller använda Environment?
   - Recommendation: Binding-approach (matchar ConversationListView exakt). AppRouter äger state, skickar till MainTabView, som skickar till FriendListView. Konsekvent mönster.
