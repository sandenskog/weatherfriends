# Phase 4.4: Fixa authUid-population vid Friend-skapande - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Alla Friend-dokument får korrekt authUid via lookup mot users-kollektionen — så att chatt fungerar för alla vänner. Tre skrivvägar ska fixas: AddFriendSheet, ContactImportService.saveReviewedContacts(), och OnboardingViewModel.completeOnboarding(). Ingen ny capability — enbart fixa befintliga kodvägar.

</domain>

<decisions>
## Implementation Decisions

### Matchningslogik
- Claude väljer bästa matchningsmetod per kodväg baserat på tillgänglig data
- Claude avgör om lookup ska vara klient-side (Firestore-query) eller server-side (Cloud Function)
- Claude väljer om en gemensam metod eller separata per kodväg är bäst
- Claude avgör om retry/re-lookup vid nil behövs eller om engångs-lookup räcker

### Retroaktiv fix
- Appen har bara testdata — ingen kritisk migration behövs
- Claude avgör om en migreringsfunktion ska byggas eller om det räcker att fixa skrivvägarna

### UX vid ingen match
- Claude väljer hur vänner utan authUid visas i NewConversationSheet (dölj, gråa ut, eller befintligt beteende)
- Claude avgör om felmeddelandet "Den här vännen har inget konto i appen än" ska justeras
- Claude avgör om feedback ska ges vid tillägg av vän utan match
- Auto-uppdatering av authUid vid ny user-registrering: Claude avgör om det ryms i fasen eller ska deferras

### Claude's Discretion
Användaren gav Claude full diskretion på samtliga beslut i denna fas. De tekniska valen och UX-detaljerna avgörs under research och planering. Fokus bör vara på att lösa gapet (Friend.authUid = nil) med enklaste möjliga approach som uppfyller alla success criteria.

</decisions>

<specifics>
## Specific Ideas

Inga specifika krav — öppet för standardapproach. Målet är att chat-funktionaliteten (direktchatt + gruppchattar) ska fungera för alla vänner som har matchande Auth-konto.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `FriendService` (Services/FriendService.swift): Har `addFriend(uid:friend:)` och `updateFriend(uid:friendId:data:)` — naturligt ställe för lookup-logik
- `UserService` (Services/UserService.swift): Har `fetchUser(uid:)` — kan utökas med sökmetoder
- `Friend.authUid` (Models/Friend.swift): Fältet finns redan som `String?`, inga modelländringar behövs
- `AppUser` (Models/AppUser.swift): Document ID = Firebase Auth UID, har `displayName`, `city`, `cityLatitude/Longitude`

### Established Patterns
- Friend-skapande går alltid via `FriendService.addFriend(uid:friend:)` — centraliserad skrivpunkt
- Alla tre kodvägar bygger `Friend(...)` utan authUid och anropar `friendService.addFriend()`
- ContactImportService har tillgång till telefon/e-post från kontakter (ej i Firestore dock)

### Integration Points
- `AddFriendSheet.addFriend()` (rad 155-176): Bygger Friend med namn+stad, anropar friendService
- `ContactImportService.saveReviewedContacts()` (rad 214-239): Bygger Friend med namn+stad per contact
- `OnboardingViewModel.completeOnboarding()` (rad 122-133): Bygger Friend från pendingFriends
- `NewConversationSheet` (rad 221-224): Guardar på `friend.authUid` — mottagare av fixen

</code_context>
