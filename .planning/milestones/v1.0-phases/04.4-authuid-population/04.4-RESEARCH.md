## Summary

Problemet är precist avgränsat: fas 4.2 lade till `Friend.authUid: String?` och uppdaterade läslogiken i `NewConversationSheet` att kräva det — men **ingen** av de tre kodvägar som skapar `Friend`-dokument sätter fältet. Alla vänner i Firestore har därför `authUid == nil`, vilket gör att direktchatt alltid misslyckas och gruppchattar är omöjliga att skapa.

Matchningslogiken är enkel: `AppUser`-dokumentens document ID i Firestore är exakt Firebase Auth UID. Det finns ingen sekundär identifierare (e-post, telefon) lagrad i `users`-kollektionen att matcha mot. Den enda matchningsbart data som finns i Friend-skrivvägarna är `displayName`. En displayName-query är opålitlig (duplikat, stavningsskillnader) och ska inte användas. Istället är rätt ansats att vid AddFriendSheet låta användaren söka och välja bland registrerade användare direkt — men det är en kapabilitet bortom fasens scope. Fasens rätta scope är att **vid Friend-skapande göra ett enkelt displayName-baserat lookup-försök** och, om det lyckas, skriva authUid. Matchar inget är Friend fortfarande skapbar utan authUid — vilket är det korrekt hanterade fallet för vänner som inte har app-konto.

**Primary recommendation:** Lägg till en `lookupAuthUid(byDisplayName:)` metod i `UserService` som gör en Firestore `whereField("displayName", isEqualTo:)`-query mot `users`-kollektionen. Anropa den i alla tre skrivvägar innan `friendService.addFriend()`. Resultatet (String? — nil om ingen match) skrivs in i `Friend.authUid`. Denna approach är helt klient-side, kräver ingen Cloud Function, och hanterar no-match-fallet naturligt.

## Architecture Patterns

### Recommended Project Structure

Inga nya filer behövs. Ändringar görs i:

```

│   └── FriendService.swift        # Oförändrad
├── Features/

│   └── Onboarding/
│       └── OnboardingViewModel.swift  # +authUid-lookup i completeOnboarding()
└── Services/
    └── ContactImportService.swift     # +authUid-lookup i saveReviewedContacts()
```

### Pattern 1: UserService.lookupAuthUid

**What:** En ny metod i `UserService` som tar `displayName: String` och returnerar `String?` (Firebase Auth UID om match, nil annars).

**When to use:** Anropas av alla tre Friend-skrivvägar direkt innan `friendService.addFriend()`.

**Example:**

```swift
// I UserService.swift
func lookupAuthUid(byDisplayName displayName: String) async -> String? {
    let snapshot = try? await db
        .collection("users")
        .whereField("displayName", isEqualTo: displayName)
        .limit(to: 1)
        .getDocuments()
    return snapshot?.documents.first?.documentID
}
```

Notera: returtypen är `String?` (inte `throws`) — ingen match är ett normalt utfall, inte ett fel. `try?` absorberar nätverksfel — returnerar nil vid timeout/fel vilket är safe: vännen skapas utan authUid och kan chattas med när authUid fixas retroaktivt.

### Pattern 2: Lookup i AddFriendSheet

**What:** I `AddFriendSheet.addFriend()` — lägg till lookup mellan Friend-byggande och `friendService.addFriend()`.

**Example:**

```swift
// I AddFriendSheet.swift — addFriend()
private func addFriend() async {
    isSaving = true
    defer { isSaving = false }

    let resolvedAuthUid = await userService.lookupAuthUid(
        byDisplayName: name.trimmingCharacters(in: .whitespacesAndNewlines)
    )

    let friend = Friend(
        authUid: resolvedAuthUid,         // nil om ingen match
        displayName: name.trimmingCharacters(in: .whitespacesAndNewlines),
        photoURL: nil,
        city: city,
        cityLatitude: latitude,
        cityLongitude: longitude,
        isFavorite: false,
        isDemo: false
    )

    do {
        try await friendService.addFriend(uid: uid, friend: friend)
        onAdded()
        dismiss()
    } catch {
        errorMessage = "Kunde inte lägga till vän: \(error.localizedDescription)"
    }
}
```

`AddFriendSheet` tar idag `uid: String` och `friendService: FriendService` via `let`. `UserService` behöver injiceras — enklaste approach är `@Environment(UserService.self)` konsekvent med projektets mönster (UserService injiceras i rotnivå via `HotAndColdFriendsApp`).

### Pattern 3: Lookup i ContactImportService.saveReviewedContacts

**What:** Varje `ReviewedContact` har `contact.fullName` — använd det som displayName vid lookup.

**Example:**

```swift
// I ContactImportService.swift — saveReviewedContacts()
for reviewed in reviewedContacts {
    let resolvedAuthUid = await userService.lookupAuthUid(
        byDisplayName: reviewed.contact.fullName
    )

    let cityDisplay = reviewed.city.isEmpty
        ? "Okänd plats"
        : (reviewed.country.isEmpty ? reviewed.city : "\(reviewed.city), \(reviewed.country)")

    let friend = Friend(
        authUid: resolvedAuthUid,
        displayName: reviewed.contact.fullName,
        photoURL: nil,
        city: cityDisplay,
        cityLatitude: reviewed.latitude,
        cityLongitude: reviewed.longitude,
        isFavorite: false,
        isDemo: false
    )

    try await friendService.addFriend(uid: uid, friend: friend)
}
```

`ContactImportService` är `@Observable @MainActor` och tar redan `friendService: FriendService` som parameter. `UserService` ska läggas till som ny parameter på `saveReviewedContacts(uid:reviewedContacts:friendService:userService:)`. Alla anropssiter (`ImportReviewView.saveAll()`) måste uppdateras att skicka med `userService`.

### Pattern 4: Lookup i OnboardingViewModel.completeOnboarding

**What:** `pendingFriends` är `[PendingFriend]` med `displayName` — lookup per friend.

**Example:**

```swift
// I OnboardingViewModel.swift — completeOnboarding(), steg 6
for (index, pending) in pendingFriends.enumerated() {
    let resolvedAuthUid = await userService.lookupAuthUid(
        byDisplayName: pending.displayName
    )

    let friend = Friend(
        authUid: resolvedAuthUid,
        displayName: pending.displayName,
        city: pending.city,
        cityLatitude: pending.cityLatitude,
        cityLongitude: pending.cityLongitude,
        isFavorite: index < 6,
        isDemo: false
    )
    try await friendService.addFriend(uid: uid, friend: friend)
}
```

`completeOnboarding(uid:authManager:userService:friendService:)` tar redan `userService: UserService` — ingen signaturändring behövs.

### Anti-Patterns to Avoid

- **Parallell lookup i ContactImportService:** CLGeocoder-mönstret i projektet kör sekventiellt för att undvika API-begränsningar. Firestore-queries är däremot thread-safe och tål parallellism, men sekventiell lookup är enklare och tillräckligt snabb för en importlista (inga Rate-limit-problem med Firestore reads).
- **Kasta fel vid no-match:** `lookupAuthUid` ska returnera `nil`, inte kasta. Vänner utan app-konto är ett normalt utfall och ska inte hindra Friend-skapandet.
- **Söka på annat än exakt displayName:** `isEqualTo:` matchar exakt. Fuzzy-match (contains, lowercase) kräver Algolia/Cloud Function och är overkill för detta use case.
- **Lägga lookup-logik direkt i varje skrivväg:** Centralisera i UserService för enkel testbarhet och DRY.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| AuthUid-lookup | Custom caching-lager | Direkt Firestore-query med `limit(to: 1)` | Firestore SDK hanterar caching internt via offline persistence |
| Felhantering vid no-match | Custom error-typ | `String?` return + `try?` absorbering | No-match är normalt utfall, inte exceptionsfall |

**Key insight:** Firestore `whereField("displayName", isEqualTo:).limit(to: 1)` är ett enkelt read-query som kostar 1 read-operation. Ingen server-side logik behövs.

## Common Pitfalls

### Pitfall 1: Friend-modellen saknar authUid i init-parametrar

**What goes wrong:** `Friend(displayName:..., city:..., isFavorite:..., isDemo:...)` — `authUid` saknas i anropen. Swift ger kompileringsfel vid strukturändring om `authUid` inte är optional med default-värde.

**Why it happens:** `Friend` är en `struct` med `var authUid: String?` — Swift kräver inte att den anges explicit i init om den är optional. Men alla tre kodvägar skapar `Friend(...)` med namngivna parametrar som hoppar över `authUid`.

**How to avoid:** Kontrollera att `Friend.init` accepterar `authUid: String? = nil` som parameter med defaultvärde. Om struct-initialisatorn inte har explict `init`, genererar Swift memberwise init automatiskt med samtliga properties — inklusive `authUid`. Verifiera att `Friend(authUid: resolvedAuthUid, displayName: ...)` kompilerar.

**Warning signs:** Kompileringsfel "Extra argument in call" eller "Missing argument" vid Friend-initialisering.

### Pitfall 2: UserService saknas i AddFriendSheet:s scope

**What goes wrong:** `AddFriendSheet` är en `struct View` och tar idag `friendService: FriendService` via `let`. `UserService` finns i `@Environment` i roten men är inte injicerat i `AddFriendSheet`.

**Why it happens:** `AddFriendSheet` presenteras med `.sheet` från `FriendListView` — inte via NavigationStack. `@Environment`-värden propagerar ner via SwiftUI view tree, men `@Observable`-klasser injicerade i roten är tillgängliga om de injiceras korrekt.

**How to avoid:** Lägg till `@Environment(UserService.self) private var userService` i `AddFriendSheet`. Verifiera att `UserService` injiceras som `.environment(userService)` i `HotAndColdFriendsApp` — det gör det redan (kontrollerat i STATE.md: "UserService injiceras som @Environment i appens rot").

**Warning signs:** Runtime-krasch "No value of type UserService found in environment" om UserService inte är injicerat i view-hierarkin.

### Pitfall 3: ContactImportService.saveReviewedContacts() har en befintlig signatur

**What goes wrong:** `saveReviewedContacts(uid:reviewedContacts:friendService:)` anropas från `ImportReviewView.saveAll()`. Om signaturen ändras till att ta `userService:` som ytterligare parameter måste alla anropssiter uppdateras.

**Why it happens:** Det finns bara ett anropssätt: `ImportReviewView.saveAll()` med `.standard(uid:friendService:)` mode. `ImportReviewView` har tillgång till `@Environment(UserService.self)` om den är injicerad.

**How to avoid:** Lägg till `userService: UserService` som sista parameter. Uppdatera `ImportReviewView.saveAll()` att skicka med `userService` (via `@Environment`).

**Warning signs:** Kompileringsfel "Missing argument for parameter 'userService'" om anropssiter inte uppdateras.

### Pitfall 4: Onboarding-flödet skapar vänner som PendingFriend, inte Friend

**What goes wrong:** Under onboarding samlas vänner i `pendingFriends: [PendingFriend]` och sparas *efter* att profilen skapats i steg 6 av `completeOnboarding()`. Lookup mot `users` görs med `displayName` — men under onboarding är det egna kontot just skapat och finns i Firestore. Lookup av sig själv (om man lägger till sig själv som vän) skulle ge match — men det är ett edge case som kan ignoreras.

**Why it happens:** OnboardingViewModel tar redan `userService: UserService` som parameter — ingen signaturändring behövs. Lookup anropas i befintlig loop (steg 6).

**How to avoid:** Enkel tillägg i loopen. Inga signaturändringar.

**Warning signs:** Ingen — onboarding-sökvägen är den enklaste.

### Pitfall 5: displayName-matchning är inexakt

**What goes wrong:** Om två användare har identiskt displayName, returneras den första matchande. `limit(to: 1)` väljer godtyckligt (Firestore garanterar ingen ordning utan `orderBy`).

**Why it happens:** Firestore `whereField` utan sekundär nyckel.

**How to avoid:** Acceptera detta som en känd begränsning i v1. Appen har testdata — i produktion är risken låg. Dokumenteras som en känd begränsning. En exaktare matchning (via email/telefon) kräver datalagringsändring utanför fasens scope.

**Warning signs:** Ingen synlig för användaren — fel vän får authUid. Accepterad begränsning.

## Open Questions

1. **Ska vänner utan authUid döljas eller visas med nedsatt UI i NewConversationSheet?**
   - What we know: Nuvarande beteende visar alla vänner, men tap på vän utan authUid ger felmeddelande "Den här vännen har inget konto i appen än."
   - What's unclear: Ska vänner utan authUid gråas ut (disabled appearance + tooltip) istället för att visa felmeddelande post-tap?
   - Recommendation: Behåll nuvarande beteende — felmeddelandet är tydligt och kommunicerar rätt budskap. Ingen UX-ändring behövs för att uppfylla CHAT-01/CHAT-02. Gråning är en finpolering utanför fasens scope.

2. **Ska retroaktiv migration göras för befintliga Friend-dokument med authUid == nil?**
   - What we know: Appen har bara testdata. Befintliga Friend-dokument i Firestore saknar authUid.
   - What's unclear: Räcker det att fixa framåt (nya vänner får authUid), eller ska befintliga dokument uppdateras?
   - Recommendation: Ingen migration behövs — testdata. Befintliga vänner utan authUid ger felmeddelande vid chattstart, men fixen framåt är tillräcklig för att satisfiera CHAT-01/CHAT-02 i testmiljö.

3. **Ska lookup göras parallellt för ContactImportService (många vänner)?**
   - What we know: `saveReviewedContacts()` loopar sekventiellt. Firestore-queries är async/await och thread-safe.
   - What's unclear: Om 20+ kontakter importeras, är sekventiell lookup godtagbart snabb?
   - Recommendation: Sekventiell lookup är tillräcklig för v1. Varje Firestore read tar ~50-200ms — 20 kontakter ger ~1-4 sek. Acceptabelt för en import-operation. Parallell lookup (via `withTaskGroup`) är en enkel optimering men onödig för fasens scope.
