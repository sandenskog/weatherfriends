# Phase 4.2: Chatt-UID-mismatch — Research

**Researched:** 2026-03-03
**Domain:** Firebase Firestore datamodell, Swift iOS — Friend-modell vs Auth UID
**Confidence:** HIGH (källkod direkt inspekterad)

---

## Summary

Buggen är identifierad och väldefinierad. `NewConversationSheet` skickar `friend.id` (Firestore-dokumentets auto-genererade ID i subcollektionen `users/{currentUid}/friends/{docId}`) som `friendUid` till `ChatService.getOrCreateDirectConversation()`. Det stämmer inte — `participants`-arrayen i ett konversationsdokument förutsätter Firebase Auth-UID:n, inte slumpmässiga Firestore-dokument-ID:n.

Konsekvensen är att konversationsdokumentet skapas med ett felaktigt deltagar-ID. Mottagarens `startListeningToConversations(uid:)` kör en Firestore-query `whereField("participants", arrayContains: uid)` med sin riktiga Auth UID — och träffar aldrig det felaktiga dokumentet. Konversationen är osynlig för mottagaren.

Lösningen kräver att `Friend`-modellen utökas med ett `authUid`-fält (mottagarens Firebase Auth UID), att detta fält populeras vid tillägg av vän, och att `NewConversationSheet` använder `friend.authUid` istället för `friend.id`. Fixen är strukturell men lokalt avgränsad — inga Cloud Functions eller backend-ändringar behövs.

**Primary recommendation:** Lägg till `var authUid: String?` på `Friend`-modellen. Populera fältet vid vän-tillägg om en matchande `AppUser` finns i Firestore. Använd `friend.authUid` (med fallback-guard) i `openDirectConversation(with:)`.

## Buggens rotorsak — Detaljerad Analys

### Dataflödet idag (trasigt)

```
Firestore-struktur:
  users/{currentUid}/friends/{AUTO_DOC_ID}   ← Friend.id = AUTO_DOC_ID
  users/{recipientAuthUid}/...               ← recipientAuthUid ≠ AUTO_DOC_ID

NewConversationSheet.openDirectConversation(with: friend):
  friendId = friend.id   // = AUTO_DOC_ID (t.ex. "xK2pQr7mNvL...")
  chatService.getOrCreateDirectConversation(
    currentUid: currentUid,
    friendUid: friendId   // FEL! Skickar Firestore doc-ID
  )

ChatService.getOrCreateDirectConversation:
  conversationId = [currentUid, friendId].sorted().joined(separator: "_")
  participants: [currentUid, friendId]   // friendId är doc-ID, inte Auth UID

ChatService.startListeningToConversations(uid: recipientAuthUid):
  .whereField("participants", arrayContains: recipientAuthUid)
  // recipientAuthUid ≠ friendId → ingen träff → konversationen syns ej
```

### GroupMode-buggen

I `createGroup()` används `selectedFriendIds` som byggs av `toggleSelection(friend:)`:

```swift

    guard let id = friend.id else { return }   // = Firestore doc-ID

private func createGroup() async {

        creatorUid: currentUid,
        participantUids: participantIds   // FEL!
    )
}
```

## Architecture Patterns

### Recommended Project Structure

Inga nya filer behövs. Ändringarna görs i befintliga filer:

```
HotAndColdFriends/
├── Models/
│   └── Friend.swift                    ← Lägg till authUid-fält
├── Features/
│   └── Chat/
│       └── NewConversationSheet.swift  ← Använd authUid istället för id
└── Services/
    └── FriendService.swift             ← Eventuellt hjälpmetod för authUid-lookup
```

### Pattern 1: authUid på Friend-modellen

**Vad:** Lagra mottagarens Firebase Auth UID direkt på Friend-dokumentet i Firestore.

**När:** När en vän läggs till via `AddFriendSheet` eller kontaktimport — om personen har ett konto i appen kan deras Auth UID slås upp.

**Exempel (efter fix):**

```swift
// Friend.swift
struct Friend: Codable, Identifiable {
    @DocumentID var id: String?
    var authUid: String?           // ← NYTT: Firebase Auth UID för chattmottagaren
    var displayName: String
    // ... övrigt oförändrat
}
```

```swift
// NewConversationSheet.swift — openDirectConversation
private func openDirectConversation(with friend: Friend) async {

        return

    isCreating = true
    defer { isCreating = false }
    do {
        let conversationId = try await chatService.getOrCreateDirectConversation(
            currentUid: currentUid,
            friendUid: friendAuthUid   // ← Auth UID, inte doc-ID
        )
        onConversationCreated(conversationId)
    } catch {
        errorMessage = error.localizedDescription
    }
}
```

### Pattern 2: ChatView-kompatibilitet

`ChatView` tar en `conversationId: String` och använder `ChatMessage.senderId`. `senderId` sätts till `currentUid` (Auth UID) — detta är redan korrekt. Inga ändringar behövs i ChatView eller ChatService förutom att rätt UID:n nu hamnar i `participants`.

### Anti-Patterns to Avoid

- **Använda `friend.id` (Firestore doc-ID) som participant-ID.** Firestore doc-ID:n i en subcollection är helt skilda från Firebase Auth UID:n och har ingen förutsägbar relation till dem.
- **Slå upp Auth UID i realtid vid chattstart utan att cacha.** Onödig latens och potentiell felkälla — lagra `authUid` direkt på Friend-dokumentet.
- **Kräva att vännen har ett konto för att LÄGGAS TILL.** Det är OK att ha vänner utan `authUid` — de kan bara inte chattas med via appen.

## Don't Hand-Roll

| Problem | Bygg inte | Använd istället | Varför |
|---------|-----------|-----------------|--------|
| Auth UID-lookup | Ingen egen ID-mapping-tabell | Firestore `users/{uid}` finns redan | AppUser lagras redan med Auth UID som doc-ID |
| Konversationslyssnare | Ingen ny query | Befintlig `whereField("participants", arrayContains: uid)` | Fungerar korrekt när rätt UID:n lagras |

## Common Pitfalls

### Pitfall 1: UI-checkmarks för grupval använder fel ID-set

**Vad går fel:** Checkmark-logiken i `friendRow` kontrollerar `selectedFriendIds.contains(friend.id)` men efter fix lagras Auth UID:n i `selectedFriendIds`. Om UI:t fortfarande kontrollerar `friend.id` visas ingen checkmark.

**Varför:** Dubbel referens — ett ställe lagrar ID, ett annat kontrollerar det.

**Hur undvika:** Ändra BÅDA platserna: `toggleSelection` och `friendRow`-checkmark-logiken till `friend.authUid`.

**Varningstecken:** Vänner kan väljas (tryck fungerar) men checkmark visas aldrig.

### Pitfall 2: Vänner utan authUid (lokala vänner / demo-vänner)

**Vad går fel:** Vänner tillagda manuellt via `AddFriendSheet` eller via kontaktimport har inte nödvändigtvis ett appkonto. `friend.authUid` är `nil` för dem.

**Varför:** `authUid` populeras bara när personen har ett konto i appen.

**Hur undvika:** Guard på `authUid != nil` i `openDirectConversation` med ett tydligt felmeddelande till användaren. I grupplistan kan man filtrera bort vänner utan `authUid` från `filteredFriends`.

**Varningstecken:** Krasch eller tom `conversationId` om `authUid` antas vara satt.

### Pitfall 3: Bakåtkompatibilitet — befintliga Friend-dokument

**Vad går fel:** Befintliga vän-dokument i Firestore saknar `authUid`-fältet. Codable-avkodning misslyckas om fältet är icke-optional.

**Varför:** Firestore returnerar inga extra default-värden för saknade fält.

**Hur undvika:** Deklarera som `var authUid: String?` (optional) — befintliga dokument avkodas med `nil`.

### Pitfall 4: Grupp-konversation skapas med blandning av doc-ID och Auth UID

**Vad går fel:** Om `toggleSelection` fixas men `createGroup` inte uppdateras (eller vice versa) blandas ID-typer i `participants`.

**Hur undvika:** Fixa `toggleSelection` och `createGroup` i en och samma commit.

## Code Examples

### Komplett fix — NewConversationSheet (relevanta delar)

```swift
// Source: Direkt från kodbas — NewConversationSheet.swift

// 1. toggleSelection — group mode
private func toggleSelection(friend: Friend) {
    guard let authUid = friend.authUid else { return }
    if selectedFriendIds.contains(authUid) {
        selectedFriendIds.remove(authUid)
    } else {
        selectedFriendIds.insert(authUid)
    }
}

// 2. checkmark i friendRow (group mode)
if let uid = friend.authUid, selectedFriendIds.contains(uid) {
    Image(systemName: "checkmark.circle.fill")
        .foregroundStyle(.blue)
} else {
    Image(systemName: "circle")
        .foregroundStyle(.secondary)
}

// 3. openDirectConversation
private func openDirectConversation(with friend: Friend) async {
    guard let friendAuthUid = friend.authUid, !currentUid.isEmpty else {
        errorMessage = "Den här vännen har inget konto i appen."
        return
    }
    isCreating = true
    defer { isCreating = false }
    do {
        let conversationId = try await chatService.getOrCreateDirectConversation(
            currentUid: currentUid,
            friendUid: friendAuthUid
        )
        onConversationCreated(conversationId)
    } catch {
        errorMessage = error.localizedDescription
    }
}

// 4. createGroup — participantIds är nu Auth UID:n via selectedFriendIds
private func createGroup() async {
    guard !currentUid.isEmpty, canCreateGroup else { return }
    isCreating = true
    defer { isCreating = false }
    let participantIds = Array(selectedFriendIds)  // Nu Auth UID:n
    let name = groupName.trimmingCharacters(in: .whitespacesAndNewlines)
    do {
        let conversationId = try await chatService.createGroupConversation(
            creatorUid: currentUid,
            participantUids: participantIds,
            groupName: name.isEmpty ? nil : name
        )
        onConversationCreated(conversationId)
    } catch {
        errorMessage = error.localizedDescription
    }
}
```

## Open Questions

1. **Ska vänner utan authUid filtreras bort i chattlistan?**
   - Vad vi vet: `filteredFriends` i NewConversationSheet returnerar alla vänner inkl. lokala.
   - Vad som är oklart: Ska lokala vänner (utan appkonto) döljas i chattlistan, eller visas med disabled/grayed-out state?
   - Recommendation: Visa grayed-out med en förklarande label ("Inte i appen") — men för Phase 4.2 räcker det med guard + felmeddelande. UI-förfining kan göras separat.

2. **Behövs authUid-lookup vid AddFriend?**
   - Vad vi vet: `AddFriendSheet` skapar en Friend utan `authUid`. UserService.fetchUser kräver känt UID.
   - Vad som är oklart: Det finns ingen sökning på displayName/email i Firestore-structuren nu.
   - Recommendation: Populera `authUid` i ett framtida steg (t.ex. via e-postlänkning). För Phase 4.2 räcker det att fältet finns och skyddas med guard. Kontaktimport via `saveReviewedContacts` kan eventuellt koppla `authUid` i en framtida fas.
