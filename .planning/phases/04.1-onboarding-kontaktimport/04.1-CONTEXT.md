# Phase 4.1: Fixa onboarding-kontaktimport - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Kontakter importerade via onboarding-flödet ska få AI-platsgissning (guessLocations()) och visas i ImportReviewView med konfidens-färger och korrigeringsmöjlighet innan de sparas. Alla sparade vänner ska ha cityLatitude/cityLongitude så att WeatherKit kan returnera väderdata.

</domain>

<decisions>
## Implementation Decisions

### Onboarding-flöde
- Återanvänd befintliga ImportReviewView — anpassa för PendingFriend istället för direkt Firestore-sparning
- Flödet blir: Välj kontakter → "Analyserar..." spinner i knappen → ImportReviewView (sheet) → Godkänn → Kontakter tillbaka som PendingFriends med stad + koordinater
- Loading-indikator i samma vy (spinner + "Analyserar..." i toolbar-knappen) — samma mönster som ContactImportView
- Kontakter sparas som PendingFriends (med koordinater) tillbaka i onboarding-listan — sparas till Firestore först vid "Slutför"

### Koordinat-upplösning
- Auto-geocoding via MapKit när ImportReviewView bygger review-items från AI-gissningar
- Geocoding sker vid review-byggande (innan review-vyn visas) — användaren ser koordinat-status direkt
- Kontakter där geocoding misslyckas markeras som "behöver korrigeras" (röd prick) — användaren kan manuellt söka stad
- Fixa BÅDA flödena: auto-geocoding i ImportReviewView.buildReviewItems() gäller oavsett varifrån review-vyn öppnas (onboarding och vanligt ContactImportView)

### Felhantering
- Om guessLocations() misslyckas (nätverksfel, Cloud Function nere): visa ImportReviewView ändå med tomma gissningar — användaren kan manuellt söka stad per kontakt. Samma beteende som befintligt flöde.

### Claude's Discretion
- Exakt refaktorering av ImportReviewView för att stödja både PendingFriend-läge och direkt-Firestore-läge
- Geocoding-batchning och throttling (MapKit rate limits)
- Hur koordinat-status presenteras visuellt i review-vyn

</decisions>

<specifics>
## Specific Ideas

- ImportReviewView behöver anpassas med ett "läge" — onboarding (returnerar PendingFriends via callback) vs. standard (sparar direkt till Firestore med uid)
- Auto-geocoding löser ett problem som finns i BÅDA flödena idag — inte bara onboarding

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ImportReviewView` (ImportReviewView.swift): Komplett review-vy med konfidens-färger, inline stad-korrigering via MapKit. Behöver anpassas för PendingFriend-mode.
- `ContactImportService.guessLocations()` (ContactImportService.swift:207): AI-platsgissning via Cloud Function — finns och fungerar
- `LocationService` (används i flera vyer): MapKit-baserad stad-autocomplete och geocoding — `resolveLocation()` returnerar CLPlacemark med koordinater
- `ContactImportOnboardingWrapper` (OnboardingFavoritesView.swift:351): Nuvarande wrapper — behöver utökas med guessLocations() och ImportReviewView

### Established Patterns
- Sheet-presentation: ContactImportView öppnar ImportReviewView som `.sheet`
- Loading i toolbar: "Analyserar..." spinner i confirmation-action toolbar-knapp
- Felhantering: Vid CF-fel → visa review utan gissningar (locationGuesses = [])
- PendingFriend-mönster: Onboarding samlar vänner i minnet, sparar batch vid "Slutför"

### Integration Points
- `ContactImportOnboardingWrapper` → behöver anropa `guessLocations()` och visa `ImportReviewView`
- `ImportReviewView` → behöver stödja callback-mode (PendingFriend) utöver direkt-sparning (uid + FriendService)
- `ImportReviewView.buildReviewItems()` → behöver auto-geocoding med `LocationService.resolveLocation()` / `CLGeocoder`
